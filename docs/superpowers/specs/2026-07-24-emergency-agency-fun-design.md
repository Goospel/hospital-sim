# 설계 스펙 — 재미 개선 3종: 응급 결정화 · 대가 가시화 · 폐업

> **상태**: 확정 설계 (2026-07-24, brainstorming). 다음 = writing-plans로 구현 계획.
> **맥락**: 사용자 방향 — *"의미는 좋은데 재미가 없다. 재미가 1순위다."* 플레이테스트 + 코드 전수 분석의 진단: **실질 결정이 전부 개원 위저드에 몰려 있고, 7일 루프는 그 결과의 결정론적 재생(관람)이다.** 응급이라는 감정적 핵심에 플레이어 개입이 0이고, 시간 압박·실패 상태·변주가 없다.
> **원칙**: *"벽은 코드가 세우되, 그 벽 앞에서 괴로운 버튼은 플레이어가 직접 누르게 하라."* 판정=결정론 코드 / 해석 카피 0(show-don't-tell) 원칙은 그대로 유지한다.
> **관련**: [2026-07-19 응급 재설계 스펙](2026-07-19-received-side-emergency-redesign-design.md) — 그 §10 확장점("받는 콜 골든타임 압박 재도입")을 이 스펙이 소화한다.

## 2. 응급 결정 카드 — "벽은 코드가, 버튼은 플레이어가"

지금은 응급 콜을 `decide`가 `accept` 인자를 무시하고 자동 판정한다 — 게임 논지("당신이 그 벽이다")의 그 순간에 플레이어는 로그를 지켜볼 뿐이다. 이를 뒤집는다:

- **`needsDecision(call)` 확장**: 응급 6종(`STEMI`·`OBSTETRIC_EMERGENCY`·`NEURO_EMERGENCY`·`TRAUMA_EMERGENCY`·`ABDOMINAL_EMERGENCY`·`MEDICAL_EMERGENCY`)도 true. 하루 결정 밀도: 예약 ~5건 + 응급 2~4건 = **7~9건**. `COSMETIC_WALKIN` 자동 수용은 유지.
- **`decide(state, accept: boolean)` → `decide(state, action: DecisionAction)`**:

```ts
type DecisionAction = 'ACCEPT' | 'DECLINE' | 'BUMP_ACCEPT' | 'TIMEOUT'
```

| 액션 | 하드락일 때 | 하드락 아닐 때 |
|---|---|---|
| `ACCEPT` | 무효 — 하드락 거절로 확정(구조가 이긴다). UI는 버튼 비활성이라 실제로 못 누름 | 수용(현행 수용 경로) |
| `DECLINE` | 하드락 거절(사유=하드락) | **신규** — 자발적 거절. disposition `CHOICE`, accepted=false. 이튿날 신문에 실린다 |
| `BUMP_ACCEPT` | 무효 | §3 트레이드오프 — 조건 충족 시만 |
| `TIMEOUT` | 하드락 거절(사유=하드락 유지 — 구조가 막은 게 진실) | 거절, 사유 신설 `UNANSWERED`(응답 없음 — 환자는 다른 병원으로) |

- **하드락이어도 카드는 뜬다.** [받기] 비활성 + 사유 배지(기존 `RejectionReason` 라벨 재사용), 활성 버튼은 [돌려보내기]뿐 — 결과는 코드가 정했지만 **누르는 건 플레이어다**. 이 스펙의 핵심 장치.
- **로그·신문의 구분 유지**: "구조가 막았다(하드락)" / "내가 보냈다(거절)" / "응답 없음(UNANSWERED)"이 `unacceptedGroups`·`news.ts`에서 구분된다. 신문은 지금 못 받은 응급(하드락)만 기사화 → **플레이어 거절·UNANSWERED 응급도 기사화**로 확장(사유별 문구 차등, `FORBIDDEN_REAL_EVENT_TOKENS` 가드 유지).

## 3. 트레이드오프 — 「예약 미루고 받기」

- **조건**: 그 과 자유 의사가 0이고, 점유 원인 중 **선택진료가 있을 때만** 제3 버튼이 열린다. 점유 원인이 전부 응급이면 안 열린다(응급은 응급을 밀어내지 않는다).
- **효과**: 밀어낼 의사(결정론 규칙 — 가장 일찍 풀리는 의사)의 진행 중 예약진료를 중단한다: `busyUntil` 해제 → 그 콜의 수익 회수(`netProfitDelta −= callDelta`) → 로그에 중단(`BUMPED`) 기록 → 응급을 수용하고 그 의사를 재점유.
- **코어 전제 — 점유 원인 추적**: 현재 `busyUntil: Record<doctorId, number>`는 "언제까지"만 안다. **"무엇 때문에"를 추가**한다:

```ts
busyWith: Record<string /*doctorId*/, { callId: string; kind: CallKind; deltaManwon: number }>
```

- UI 버튼 라벨은 사실만: 「예약 미루고 받기 — 김민준 진료 중단 · −10만원」.

## 4. 카운트다운 — 응급만 실초 15초

- 응급 카드에 **실제 시간 15초** 카운트다운 바(구급대 인내 게이지). 만료 시 UI 레이어(`useHospitalClock` 계층)가 `TIMEOUT` 액션을 디스패치한다.
- **결정론 경계**: 코어(`decide`)는 실시간을 모른다 — 타임아웃도 명시 액션으로 받는다. 카운트다운 상수(`EMERGENCY_DECISION_SECONDS = 15`)는 UI 상수로 두고, vitest 결정론에 실시간이 새지 않는다.
- 예약 카드는 현행 유지(카드 중 시계 정지·무압박).

## 5. 대가 가시화 — 결정을 정보 있는 도박으로

- **예약 카드에 점유 미리보기**: 수락 결과를 사실로 표기 — 「최하은(순환기) 14:20까지 점유」. 그 시간에 순환기 응급이 오면 못 받는다는 결론은 플레이어가 스스로 잇는다(해석 카피 없음).
- **[받기] 비활성 사유 표시**: 「순환기 의사 모두 진료 중 · 16:40에 자리」. 현재는 무표시(플레이테스트에서 실측한 구멍). `startMinFor`가 이미 계산하는 값이라 **표시만 추가**한다.
- 응급 카드의 하드락 사유 배지는 §2에 포함.

## 6. 폐업 게임오버 — 무한 주차 + 파산

- 급여·고정비는 이미 `completeWeek`의 `weekNet` 경유로 금고에 반영된다 — **경제 배선 무변경**, 판정만 추가:
  - `SessionState.insolvencyStreak: number` 신설. `completeWeek`에서 갱신 후 금고 음수면 `+1`, 흑자면 `0` 리셋.
  - streak 1 → `WEEK_SUMMARY`에 은행 통지(사실 서술: 「잔고 −N만원. 다음 주도 음수면 폐업」).
  - streak 2 → `nextWeek` 봉쇄(throw), **강제 에필로그(폐업)**. `buildEpilogue`에 `closed: boolean` — 머리글만 「폐업」, 내용은 기존 누적 결산 그대로(해석 카피 추가 없음).
- 주차 무한 유지 — 유한 회차("4주 공략")를 만들지 않아 정답-퍼즐화를 피한다.

## 7. 테스트 전략 (TDD Red→Green)

- `receiving.test.ts`: 액션 4종 — 응급 `ACCEPT` 수용 / `DECLINE` 거절 기록 / 하드락에서 `ACCEPT` 무효(하드락 거절 유지) / `BUMP_ACCEPT` 점유 해제·수익 회수·응급 수용 / 점유 원인이 응급이면 BUMP 무효 / `TIMEOUT` → `UNANSWERED`(하드락이면 하드락 사유 유지).
- `session.test.ts`: `insolvencyStreak` 전이(음수 1주 경고 → 연속 2주 폐업 → 흑자 복귀 리셋) / 폐업 상태에서 `nextWeek` throw / 강제 에필로그에 `closed` 플래그.
- `news.test.ts`: 플레이어 거절·UNANSWERED 응급 기사화 + 실명 토큰 가드 유지.
- 카운트다운은 UI 훅 테스트 최소(만료 시 `TIMEOUT` 디스패치 1건).
- 게이트: 전체 vitest green + `tsc --noEmit` 0.

## 9. 리스크 (정직하게)

- **메시지 역전 감시**: 플레이어에게 선택권을 열수록 "잘 고르면 이긴다"로 읽힐 수 있다. 방어선 = 하드락은 여전히 절대(설득·선택 무관), BUMP는 수익 손실을 동반, 폐업 압박이 "옳은 선택"을 계속 비싸게 만든다 — 정답 없는 구조는 유지된다.
- **밸런스 미확정**: 응급 밀도(2~4건)·15초·연속 2주 기준은 첫 구현 값이다. 완주 플레이테스트로 조정한다(조정은 상수 변경만으로 가능하게 설계).
- **마감(2026-08-10)**: PR 3개 순차라 어느 지점에서 멈춰도 제출 가능 — PR 1만으로도 "관람 → 플레이" 전환이 성립한다.

## 10. 확장점 (⏸ 이번 스코프 밖)

- 하루 압축·페이싱(워크인 로그 노이즈 축소) — 진단 4번.
- 콜 구성 시드 랜덤화·에스컬레이션·희귀 사건 — 진단 5번.
