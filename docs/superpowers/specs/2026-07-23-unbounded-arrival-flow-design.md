## 1. 배경 — 이건 새 기능이 아니라 미완의 완성이다

[시간 기반 하루 루프 스펙](2026-07-20-time-based-day-loop-design.md)의 **목표 1번**이 이미 이것이었다:

> **하루의 단위를 개수에서 시간으로.** `createCallQueue`의 고정 5통을 시계(9:00→마감)로 교체. **콜 총량은 진료 소요시간에 따라 창발한다.**

시계·소요시간·점유·boarding 이월은 다 구현됐는데 **총량만 개수로 남았다**. `callsForBeds(beds) = beds + 2`가 하루를 5~9통으로 잠그고, 하루의 끝은 여전히 `index >= queue.length`다. 사용자 지적(2026-07-23): *"시간 시스템을 바꿨는데 콜 시스템도 그대로다. 이 둘은 공존할 수 없다."* 정확하다.

### 세 증상, 한 뿌리

| 증상 | 근인 |
|---|---|
| 하루에 받을 콜 수가 정해져 있다 | 하루의 끝 = `index >= queue.length` (개수) |
| 손님이 순서대로만 온다 | `arrivalMinFor`가 하루 600분을 콜 수로 **균등 분할** → 간격이 항상 ~120분, 절대 안 겹침 |
| 병원이 북적거리지 않는다 | 맵에 그려지는 환자 = **진료 중인 의사 수** → 의사 3명이면 화면에 환자 최대 3명 |

셋 다 "하루의 축이 콜 개수"라는 한 뿌리다. 특히 세 번째가 결정적이다 — 도착 순간 그 과 의사가 바쁘면 `NO_FREE_SPECIALIST`로 **즉시 지워지므로**, 뭉쳐서 와도 대기실에 사람이 남지 않는다. 대기라는 상태 자체가 모델에 없다.

## 2. 확정 설계 결정 (브레인스토밍 3갈래)

| 갈림 | 결정 | 기각한 것 |
|---|---|---|
| 밀려들 때 플레이어의 손 | **카드를 없애고 흐름을 본다** — 진짜 결정이 뜰 때만 멈춘다 | 통 수만 늘리기(「계속」 클릭이 4배) · 트리아지(새 시스템, 마감 대비 과대) |
| 못 받는 환자 | **기다린다 · 버티면 받고 못 버티면 떠난다** | 즉시 거절(대기열이 안 생겨 북적임 불가) · 이탈 없는 무한 대기(대가가 없어 긴장 0) |
| 늘어나는 환자의 정체 | **외래·워크인 대량 + 응급은 지금 수준** | 응급 비례 증가(신문·소송 폭증, 콜 델타가 구조 손익 압도) |

세 번째 결정이 이 게임의 논지와 맞물린다: **북적이는데 정작 응급은 뒤로 밀린다.** 미용·검진으로 복도가 가득 차고, 그 사이 STEMI가 기다리다 떠난다. 해석 카피 없이 그 광경만 놓는다(메모 `game-show-dont-tell`).

## 3. 큐 — 두 스트림

| 스트림 | 통수 | 출처 | 성격 |
|---|---|---|---|
| **응급** | 하루 2~4통 (지금 그대로) | `DAY_PLANS`의 응급 엔트리 | 배후과 요구 · 자동 판정 |
| **외래** | 하루 `beds × 20` = **60 / 100 / 140** | 워크인 위주 + 12통당 예약 1 | 자동 접수(워크인) / 플레이어 결정(예약) |

하루 총 **62 / 102 / 142명**(병상 티어 3·5·7). 개수가 상한이 아니라 **09:00~19:00라는 시각이 상한**이다. 큐가 하루를 채우므로 큐 소진 = 시간 소진이고, `callsForBeds`(=`beds+2`)는 폐기한다.

**왜 병상 비례인가**: 큰 병원일수록 더 많이 온다. 능력이 아니라 규모가 도착량을 정한다 — 능력에 비례시키면 압박이 구조적으로 사라진다(항상 감당 가능해진다).

**왜 20배인가 — 북적임은 뭉침이 아니라 도착률이 만든다.** 초안은 `beds × 5`(하루 17명)였는데 브라우저에서 **여전히 한산했다**(실측: 대기 1명, 의사 4명 중 3명 자유). 리틀의 법칙이 정직하다 — 35분에 한 명 오고 진료가 45분이면 동시 인원이 **1.3명**이다. 도착을 아무리 뭉치게 해도 그 평균은 못 넘는다(뭉침은 분산이지 평균이 아니다). `× 20`이면 동시 5~12명이 되고, 수익과를 꽉 채웠을 때(미용 3 + 검진 3 = 3600분 ÷ 45분 ≈ 80통) 겨우 감당하는 수준이라 **채용 선택이 처음으로 처리량을 가른다**.

**응급을 안 늘리는 이유**: 응급 전원 콜은 현실에서도 하루 몇 통이다. 북적임을 만드는 건 외래다. 그리고 응급을 4배로 늘리면 `turnedAway` → 신문 헤드라인이 4배가 되어 결말 화면이 마비되고, 소송 노출도 함께 4배가 된다.

## 4. 도착 — 뭉침

```
지금: arrivalMinFor(w, d, i, count) = i × (600/count) + jitter(0..600/count)
바꿈: arrivalMinFor(w, d, i)        = seededUnit(callSeed(w, d, i, 2)) × 600
```

`count` 인자가 사라진다. 하루 전체에 균등 랜덤으로 뿌리면 **간격이 지수분포에 가까워져 자연히 몰리고 빈다** — 뭉침을 위해 별도 피크 모델(아침·점심 러시)을 만들지 않는다. `createCallQueue`가 이미 마지막에 도착순 정렬을 하므로 정렬 로직은 그대로다.

결정론은 유지된다(`seededUnit` + `callSeed` 파생, `Math.random` 0건).

**파급**: `nightShift`는 이미 `arrivalMin >= NIGHT_START_MIN`에서 파생하므로 자동으로 따라온다. 다만 `DAY_PLANS` 주석의 *"count=5 슬롯에선 항상 마지막 1통만 야간"* 은 더 이상 참이 아니다 — 야간 콜 수가 날마다 다른 게 오히려 자연스럽다. 그 주석을 고친다.

## 5. 대기 — `decide`의 한 갈래

지금 `decide`는 도착 시각에 그 과 자유 의사가 없으면 그 자리에서 못 받는 걸로 끝난다. 여기에 **기다림**을 넣는다.

```
도착(arrivalMin)에 그 과 자유 의사 있음  → 즉시 시작 (지금과 동일)
                          없음  → 가장 빨리 비는 의사의 시각까지 대기
                                  ├ 대기시간 ≤ 한계 → 그 시각에 시작
                                  └ 대기시간 >  한계 → 이탈 (LEFT_WAITING)
그 과 의사가 아예 없음(미채용·배후 없음) → 지금처럼 즉시 하드락 (기다려도 소용없다)
```

### 5.1 대기 한계 (`PATIENCE_MIN`)

각색값이되 **상대 길이만** 의미가 있다(임상 주장 아님).

| 종목 | 한계 | 근거 |
|---|---|---|
| `STEMI` | 90분 | 재관류 골든타임(medical-system-grounding.md) |
| `OBSTETRIC` / `NEURO` / `TRAUMA` / `ABDOMINAL` | 120분 | 응급 수술급 |
| `MEDICAL_EMERGENCY` | 180분 | 고열·감염 — 덜 급하다 |
| `COSMETIC_WALKIN` | 60분 | 기다리다 그냥 간다 |
| `SPECIALIST_ELECTIVE` | 120분 | 예약이라 어느 정도 기다린다 |

### 5.2 대기 순서 — 도착순(FIFO), 트리아지 없음

같은 과 의사를 여럿이 기다리면 **도착순**으로 받는다. 트리아지를 넣지 않는 건 게으름이 아니라 논지다: **미용 환자가 응급보다 먼저 자리를 먹는 일이 실제로 벌어진다.** 다만 과가 다르면 애초에 경쟁하지 않으므로(미용 의사 ≠ 순환기 의사), 실제 충돌은 `SPECIALIST_ELECTIVE` ↔ 같은 과 응급뿐이다 — 그건 [시간 스펙](2026-07-20-time-based-day-loop-design.md)이 이미 설계한 갈등이고, 이번에 그 갈등이 **잦아진다**.

### 5.3 새 사유 태그 `LEFT_WAITING`

`RejectionReason`에 추가한다. 기존 4종이 "구조가 막았다"라면 이건 **"구조가 늦었다"**다 — 뺑뺑이의 실제 형상이라 별도 태그가 맞다. 응급이 이탈하면 기존 `turnedAway` 경로를 그대로 타 다음날 신문이 된다([news.ts](../../../src/game/news.ts)의 `REASON_CLAUSE`에 문구 추가).

## 6. 북적임 — 맵의 대기 환자

`log` 엔트리에 **`startMin`**(진료 시작 시각)을 싣는다. 그러면 시각 `atMin`에서:

```
arrivalMin ≤ atMin < startMin   → 지금 대기 중 (복도에 선다)
startMin   ≤ atMin < endMin     → 진료 중 (지금도 그린다)
이탈한 사람                      → startMin 없음, arrivalMin..이탈시각 동안 대기로 보이다 사라진다
```

**새 게임 상태 0개** — `deriveMapScene`이 `log` + `queue`에서 파생만 한다([맵 스펙](2026-07-23-hospital-map-design.md)의 0 침습 원칙 유지). 대기 환자는 복도 존(`CORRIDOR`)에 서고, 인원이 늘면 기존 `positionOf`의 간격 축소 로직이 이미 화면 밖 이탈을 막는다.

⚠️ 복도 폭은 유한하다. 대기 인원이 10명을 넘으면 스프라이트가 겹치므로, **표시 상한을 두고 넘치는 인원은 「+N」 배지**로 처리한다. 게임 상태는 상한을 모른다(표시 전용).

## 7. 흐름 — 카드 제거

지금은 콜마다 카드가 뜨고 플레이어가 버튼을 누른다. 하지만 **실제 결정은 `SPECIALIST_ELECTIVE` 하나뿐**이다(응급은 `decide`가 `accept`를 무시하고 자동 판정, 워크인은 `isAutoAccept`로 자동 접수). 통수가 4배가 되면 「계속」 클릭만 4배가 된다.

```
흐름 중 → 시계가 콜의 arrivalMin을 지나면 그 콜을 자동 처리(decide)
        → SPECIALIST_ELECTIVE에 닿으면 멈추고 카드
        → 마지막 콜 이후 dayEndMin까지 흐른 뒤 마감
```

`useHospitalClock`의 `flowTargetMin`을 **"다음 콜"에서 "다음 결정 필요 콜"**로 바꾸면 된다. HUD의 `콜 3 / 5` 카운터는 제거하고 **시각 · 대기 N명 · 오늘 받은 수**로 바꾼다 — 그 카운터가 "제한이 있다"는 인상의 직접적 출처다.

## 8. 경제 — 통수가 는 종목만 손댄다

외래가 12배가 되면 콜 델타 누적이 구조 손익을 압도한다(불변식 **I8: |순이익| ≤ 4 × 예산 = 400억**). [receiving.ts의 스케일 경고](../../../src/game/receiving.ts)가 이미 예고한 실패 모드다.

| 종목 | 지금 | 바꿈 | 비율 | 이유 |
|---|---|---|---|---|
| `COSMETIC_WALKIN` | 6 / 3 | **1 / 1 (델타 0)** | — | ⏸ 이중 계상 제거 — 아래 |
| `SPECIALIST_ELECTIVE` | 10 / 6 | **5 / 3** | 166.7% (검체 160.5%) | 통수 3배 → 비율 보존 축소 |
| 급여 수술 4종 + `STEMI` | 11 / 13 | **그대로** | 84.6% | 통수 안 늘었다 |
| `MEDICAL_EMERGENCY` | 3 / 6 | **그대로** | 50% | 통수 안 늘었다 |

**🔴 응급을 건드리면 안 된다.** 초안은 전 종목을 절반으로 내렸는데, 그러자 양심 경로 결말이 −3억에서 **+4억으로 뒤집혔다**(실측) — 비율(83%·50%)은 멀쩡한데 *"양심적으로 하면 적자"*라는 논지가 증발했다. 스케일 조정은 **통수가 는 종목에만** 건다. 정수 해상도도 함정이다: `1/2`는 비율 50%가 정확하지만 델타가 −1이 되어 STEMI(−1)와 같아져 불변식 I3(|STEMI| < |고열감염|)이 죽는다.

### 8.1 ⏸ 워크인 델타 0 — 이중 계상의 임시 처방

워크인이 하루 55통이 되자 **같은 진료를 두 층에서 세고 있었음**이 드러났다([T-069](../../../claude-docs/troubleshooting/T-069.md)):

- `segments`(부문 손익) = "미용 의사 1명이 주 70억 번다" — 의사 수만 보고 뭉뚱그린 값
- `CALL_ECONOMICS` = "워크인 한 명 → +1억" — 환자를 한 명씩 센 값

①의 70억이 나오는 출처가 ②가 세는 그 환자들이다. 실측으로 콜 델타가 **404억**을 더해 순이익 707억(I8의 1.77배)이 됐다. 워크인 델타를 0으로 두어 미용 흑자를 부문 손익 한 곳에서만 세게 했다(707 → **322억**).

> ✅ **해소됨 (2026-07-23, 후속 PR).** 아래 "별도 작업"이 실제로 수행됐다 — 부문 손익이 `fixedCostPerDoctorManwon`(항상 양수)로 뒤집혔고, 워크인 델타는 0에서 **+15만원**으로 되살아났다. 단위도 만원으로 내려갔다(보톡스 30만원 / 응급 PCI 850만원). 상세: [T-071](../../../claude-docs/troubleshooting/T-071.md) · [changeLog](../../../claude-docs/changeLog.md). 아래 문단은 그때의 진단으로 보존한다.

**근본 처방은 별도 작업**(사용자 결정 2026-07-23): 부문 손익을 *"그 과가 버는 돈"*에서 **"그 과의 고정비"**로 뒤집고 콜을 유일한 수익원으로 둔다. 그러면 과별 흑자·적자가 입력이 아니라 **계산에서 창발**한다 — [receiving.ts](../../../src/game/receiving.ts)의 `WORKUP_ECONOMICS` 주석이 이미 요구하는 그것이다. 그때 금액도 만원 단위 현실 수치로 되살아난다(*보톡스 한 건 1억*은 각색을 넘어 틀렸다 — 사용자 지적). 파급: `setup.ts`·`world.ts`(재정중립 패키지)·`ledger.ts`·`growth.ts`(성장 비용 재조정).

### 8.2 I8을 테스트로 못박는다

이 불변식은 그전까지 주석에만 있었고, 그 주석 자체가 *"테스트로 안 잡히고 브라우저 7일 완주로만 잡힌다"*고 적고 있었다 — 그리고 실제로 **안 잡혔다**. `session.test.ts`의 `[I8]` 블록이 수익 최대 빌드·양심 빌드 양쪽과 "콜 기여분 < 구조 기여분"을 검사한다.
