# T-057 · dev 서버를 켠 채 npm install → Turbopack이 캐시한 빌드 에러가 서버 재시작으로 안 풀린다

- **증상**: 제출 문서 PDF 파이프라인을 만들려고 `npm install --save-dev marked`를 실행했다. 설치는 성공(경고만). 그런데 켜져 있던 dev 서버의 게임 화면이 **Build Error**로 바뀌었다 — `Error evaluating Node.js code / Cannot find module '../lightningcss.win32-x64-msvc.node'`, import trace는 `src/app/layout.tsx` → 폰트 CSS 모듈. **설치한 `marked`와 아무 관계 없는 모듈**이 깨졌고, 그 뒤 `preview_stop` → `preview_start`로 **서버를 재시작해도 같은 에러**가 그대로 났다.
- **원인**: `npm install`은 의존성 트리를 다시 쓰면서 기존 패키지 디렉터리를 **잠시 지웠다 다시 깐다.** 그 창(window) 동안 실행 중이던 Turbopack이 `lightningcss`(Tailwind 4의 CSS 트랜스포머)의 네이티브 바이너리를 요구했고, 마침 그 파일이 없어 모듈 해석에 실패했다. **Turbopack은 그 실패 결과를 `.next` 빌드 캐시에 저장한다** — 그래서 설치가 끝나 파일이 제자리로 돌아온 뒤에도, 서버를 새로 띄우면 캐시가 그 실패를 그대로 되살린다. 오진을 유도하는 지점이 둘이다: (1) 에러 메시지가 `Cannot find module …node`라 **node_modules가 깨진 것처럼** 보이지만, 같은 시점에 `node -e "require('lightningcss')"`는 **정상 로드된다**(모듈은 멀쩡하다). (2) 보통 "서버 재시작"이 이런 상태를 푸는 표준 처방인데 여기선 **안 듣는다** — 상태가 프로세스가 아니라 디스크 캐시에 있기 때문.
- **해결**: dev 서버를 끄고 **`.next`를 지운 뒤** 다시 띄운다(`rm -rf .next`). 캐시가 비면 Turbopack이 현재의(정상인) node_modules로 다시 해석한다. 재설치·`npm ci`·lockfile 손질은 **불필요**했다 — node_modules는 처음부터 온전했다.
- **재발방지**: **의존성을 설치·제거하기 전에 dev 서버를 먼저 끈다.** 이미 밟았다면 진단 순서를 고정한다 — ① `node -e "require('<깨졌다는 모듈>')"`로 모듈이 진짜 없는지 먼저 가른다 → ② 로드가 되면 그건 **캐시 문제**이므로 `.next` 삭제(재설치로 시간 쓰지 말 것) → ③ 로드가 안 될 때만 재설치를 의심한다. 넓게는: **에러 메시지가 가리키는 계층(모듈 해석)과 실제 고장난 계층(빌드 캐시)이 다를 수 있다** — 메시지를 그대로 믿고 node_modules를 갈아엎으면 증상은 같은데 시간만 버린다([T-055](T-055.md)의 "게이트/실행 비대칭"과 같은 계열: 관측되는 자리와 원인의 자리가 어긋난다).
