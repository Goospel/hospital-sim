# T-047 · `StringBuilder.AppendLine`이 LF 파일에 CRLF를 섞어 혼합 EOL을 만든다

- **증상**: `rebuild-troubleshooting-index.ps1`이 허브의 목차 구간을 다시 쓴 뒤 파일을 재보니 **CRLF 26줄 + LF 34줄**이었다. 원본은 전부 LF(216줄)였다. 파일은 멀쩡히 렌더되고 `git`은 `core.autocrlf=true`라 아무 경고도 안 한다 — **무성 오염**이다. 이대로 커밋하면 한 줄도 안 바뀐 줄까지 전부 diff로 잡히는 phantom diff가 되고, `-Check`가 `생성본 != 파일`을 영구 참으로 만들어 **목차가 늘 stale하다고 거짓 보고**한다.
- **원인**: `[System.Text.StringBuilder]::AppendLine()`은 `Environment.NewLine`을 쓴다 — Windows에서 **CRLF**다. 반면 마커 바깥(사람이 쓴 부분)은 파일에서 읽어온 그대로라 LF다. 둘을 이어 붙이니 한 파일 안에서 EOL이 갈렸다. **글로벌 CLAUDE.md가 이 함정을 정확히 명시하고 있었다** — *"파일 재생성 시 BOM·EOL 바이트 보존 … 안 지키면 한 줄 고치려다 파일 전체가 phantom diff"*. 규칙을 읽고도 밟았다: `AppendLine`이 기본 습관이라 **EOL을 선택하고 있다는 자각 자체가 없었다.** 무엇을 쓸지 고민한 적이 없으니 규칙을 떠올릴 계기도 없었다.
- **해결**: 쓰기 전에 대상 파일의 EOL을 감지해(`$eol = if ($t.Contains("`r`n")) { "`r`n" } else { "`n" }`) `StringBuilder` 대신 줄 배열 + `-join $eol`로 조립했다. TDD로 잡았다 — LF 픽스처와 CRLF 픽스처 양쪽에서 "재생성 후 EOL 종류가 그대로인가"를 검사하는 테스트를 먼저 쓰고 Red를 봤다(케이스 11이 MIXED로 실패).
- **재발방지**: **PowerShell로 기존 파일을 다시 쓸 땐 EOL을 "고르는 결정"으로 취급한다** — `AppendLine`·`Out-File`·`Add-Content`는 전부 CRLF를 기본으로 낸다. 안전한 형태는 `배열 -join $eol` 뒤 `[System.IO.File]::WriteAllText`이고, `$eol`은 **원본에서 감지**한다. ⚠️ 테스트를 CRLF 픽스처로만 짜면 이 버그가 안 잡힌다 — CRLF 파일에 CRLF를 쓰면 우연히 맞아떨어진다(실제로 케이스 12·13은 통과했고 **LF 케이스만 실패**했다). **양쪽 EOL을 다 픽스처로 넣는다.** T-045와 같은 계열 — 거기선 통과 케이스만 짜서 검증이 0이어도 그린이었고, 여기선 CRLF 케이스만 짰으면 LF 오염을 못 봤다. **테스트가 한 방향만 보면 반대 방향은 없는 것과 같다.**
