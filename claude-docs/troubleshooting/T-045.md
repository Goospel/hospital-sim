# T-045 · fail-open 훅을 positive control 없이 테스트하면, 검증이 0이어도 전부 그린으로 보인다

- **증상**: `block-stale-base.ps1`(PreToolUse 하드가드)을 TDD로 만들며 시나리오 7건을 돌렸더니 **6건 PASS / 1건 FAIL**이 나왔다. "핵심 하나만 고치면 되겠다"고 읽었는데, **6건의 PASS는 전부 가짜였다.** 훅이 아무것도 읽지 못하고 fail-open(조용히 통과)한 것을, 테스트가 "훅이 올바르게 통과시켰다"로 집계하고 있었다. **두 번 연속 속았다.**
- **원인**: 훅은 설계상 **모든 예외에서 exit 0**이다(작업을 막으면 안 되니까). 그래서 무엇이 잘못되든 결과가 `allow`다 — `allow`를 기대하는 테스트만 있으면 **검증이 0이어도 초록**이다. 실제로 두 가지가 동시에 깨져 있었다: (1) 픽스처를 스크래치패드(경로 130자)에 만들어 `origin.git/objects/tmp_objdir-incoming-XXXXXX/<sha>`가 **Windows MAX_PATH(260)를 넘겨 `push`가 "Filename too long"으로 죽었고** → 픽스처가 빈 repo로 남아 훅이 "repo 아님"으로 통과. (2) 테스트가 `file_path`에 **bash 경로(`/c/Users/...`)**를 넘겨 PowerShell이 해석하지 못함 → 훅이 repo를 못 찾고 통과. 둘 다 **에러 메시지 없이** 그린이었다.
- **해결**: (1) 픽스처를 짧은 경로(`/c/Users/<user>/AppData/Local/Temp/hf`)로 옮김. (2) `cygpath -m`으로 Windows 경로 변환. (3) **픽스처 sanity check** — 생성 직후 "origin.git에 main이 3커밋인가 / work에 origin/main 추적이 있는가 / work가 아직 stale한가"를 검사하고 하나라도 어긋나면 **테스트를 돌리지 않고 abort**한다(가짜 그린을 만드느니 안 도는 게 낫다). (4) **positive control** — `deny`가 나와야 하는 케이스를 하나 두고, 그게 실패하면 마지막에 *"나머지 allow PASS 들은 훅이 fail-open 한 결과일 수 있다. 그린으로 취급하지 마라"*를 출력한다. 고친 뒤 7/7 통과.
- **재발방지**: **fail-open 컴포넌트(훅·가드·폴백)를 테스트할 땐 "차단되어야 정상"인 케이스를 반드시 하나 넣고, 그것이 곧 나머지 케이스의 유효성 조건이다.** 통과 케이스만으로 짠 스위트는 컴포넌트가 아예 없어도 초록이다 — 실제로 훅 파일이 존재하지 않을 때 6/7이 PASS였다. ⚠️ **T-044와 같은 계열이되 방향이 반대다** — 거기선 green이 **부풀려졌고**(워크트리 테스트까지 세서 510), 여기선 green이 **텅 비었다**(아무것도 검증 안 하고 6). 둘 다 "N green"이 아무 말도 하지 않는다. 관련: 반대편 함정으로 **fail-close** 훅(`.githooks/commit-msg`)은 훅이 없으면 전부 거부되므로 **통과 케이스가 positive control 역할**을 한다 — 방향만 뒤집어 같은 규칙을 적용한다.
