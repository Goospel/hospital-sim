# T-068 · 중단된 편집은 재개됐고, 그 편집의 검증은 재개되지 않았다

- **증상**: 위저드의 티어 사다리 구분자에 주석을 붙이는 한 줄짜리 편집을 했는데 페이지 전체가 500이 됐다. dev 서버 로그에만 `Expected '</', got 'ident'`.

  ```jsx
  {i > 0 && (
    {/* 사다리 구분자 — ... */}   // ← 여기가 표현식 자리라 JSX 주석이 못 온다
    <span aria-hidden className="text-on-desk/40">›</span>
  )}
  ```

  `{cond && ( ... )}`의 괄호 안은 JSX children이 아니라 **표현식** 컨텍스트다. `{/* ... */}` 주석은 children 자리에서만 유효하고, 여기선 `{}`가 빈 객체 리터럴로 파싱된다.

- **원인**: 문법 자체는 사소하다. 문제는 **왜 게이트를 안 지나고 브라우저까지 갔는가**다.

  이 편집은 **두 번 시도됐다.** 첫 시도는 `block-stale-base` 훅이 거부했다(origin/main이 1 커밋 앞서 있었다). 그래서 커밋 → rebase로 base를 정리하고, **같은 편집을 다시 적용**했다. 그리고 곧바로 브라우저 플레이 검증으로 넘어갔다.

  여기서 빠진 게 있다. 편집은 재개됐는데 **그 편집에 딸린 "게이트를 돌린다"는 계획은 재개되지 않았다.** 훅에 막힌 시점의 머릿속 상태는 "이 변경은 이미 다 정한 것"이었고, git 작업 후 돌아왔을 때 그 상태가 그대로 이어졌다 — 마치 **이미 검증한 변경을 다시 붙여넣는 것처럼** 느껴졌다. 실제로는 그 코드는 한 번도 컴파일된 적이 없었다.

  `tsc --noEmit`은 이걸 **정확히 잡는다**(재현 확인: `error TS1005: ')' expected`). `next build`도 잡는다. 둘 다 안 돌렸을 뿐이다. 직전에 본 초록 게이트는 **한 편집 전의 코드**에 대한 것이었다.

- **해결**: 주석을 `{i > 0 && (` 위(children 컨텍스트)로 옮기고, **그 편집 이후에** `tsc --noEmit` + `next build`를 다시 돌려 0을 확인했다.

- **재발방지**: **훅·권한 거부로 중단된 작업을 재개할 때, 재개되는 것은 편집만이고 그 앞뒤의 검증은 스스로 다시 세워야 한다.** 중단은 되돌리기 지점처럼 느껴지지만 실제로는 컨텍스트 스위치이고, 스위치를 건너 돌아오면 "이 변경은 검토가 끝났다"는 느낌만 살아남고 "아직 실행은 안 했다"는 사실이 떨어져 나간다. 거부 → 우회 → 재적용 뒤에는 **거부 직전에 하려던 다음 단계가 무엇이었는지**를 한 번 소리 내어 확인한다.

  그리고 **초록 게이트에는 시각이 붙어 있다.** "tsc 0"은 명제가 아니라 특정 트리 상태에 대한 관측이라, 그 뒤 편집이 하나라도 있으면 만료된다. 이건 [T-067](T-067.md)이 말한 "정직해 보이는 오답"의 시간 버전이다 — 게이트는 거짓말을 하지 않았고, 다만 **다른 코드에 대해** 참이었다.

  > ⚠️ 이 항목의 초안은 원인을 *"tsc가 SWC와 달리 이 JSX를 통과시킨다"*로 적었다. 그럴듯했지만 **틀렸다** — 확인하려고 같은 문법을 임시 파일로 재현하니 tsc가 5건의 에러로 잡았다. 게이트를 탓하기 전에 **그 게이트를 실제로 그 코드에 돌려 봤는지**부터 본다. 안 돌린 게이트의 침묵을 통과로 읽는 것이 이 함정의 본체다.
