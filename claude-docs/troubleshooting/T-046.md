# T-046 · `core.autocrlf=true`에서 `.gitattributes` 없이 셸 훅을 커밋하면, clone 시 CRLF 로 풀려 훅이 통째로 죽는다

- **증상**: `.githooks/commit-msg`를 `git add` 하자 *"LF will be replaced by CRLF the next time Git touches it"* 경고. 현재 워크트리에선 훅이 정상 작동해서 **문제가 없어 보였다.**
- **원인**: 이 머신은 `core.autocrlf=true`이고 레포에 `.gitattributes`가 **없었다.** index엔 LF로 정규화돼 들어가지만 **체크아웃 시 CRLF로 풀린다** — 즉 지금 이 워크트리에서만(Write로 직접 만든 LF 파일이라) 우연히 도는 상태였고, **fresh clone 하면 셔뱅이 `#!/bin/sh\r`가 되어 훅이 "실행 파일 없음"으로 죽는다.** 그리고 죽은 훅은 **아무 소리도 내지 않는다** — 커밋이 그냥 통과하므로 검사기가 사라진 걸 아무도 모른다.
- **해결**: `.gitattributes` 신설 — `*.sh text eol=lf` + `.githooks/** text eol=lf`. `git check-attr eol -- .githooks/commit-msg` = `lf` 확인, `git add --renormalize`로 index 재확인.
- **재발방지**: **실행되는 텍스트 파일(셸 스크립트·git 훅)을 레포에 넣을 땐 `.gitattributes`로 EOL을 못박는 것을 같은 커밋에서 한다.** Windows에서 `core.autocrlf=true`는 흔한 기본값이라 재발률이 높다. ⚠️ 위험의 성격이 T-045와 같다 — **"검사기가 없는 것"보다 "있는 줄 아는데 조용히 없는 것"이 더 나쁘다.** 훅·가드를 추가할 때마다 "이게 안 돌면 내가 알 수 있나?"를 한 번 묻는다.
