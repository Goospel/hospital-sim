# T-061 · 훅이 명령을 바꾸면 권한 규칙도 같이 바뀌어야 한다 — rtk 설치 후 모든 Bash 거부

- **증상**: 토큰 절감 도구 [rtk](https://github.com/rtk-ai/rtk)를 설치하고 `PreToolUse` 훅(`rtk hook claude`)을 `.claude/settings.local.json`에 등록한 뒤 Claude Code를 재시작했다. 그 직후 **가장 평범한 읽기 명령이 거부됐다**.
  ```
  Bash: git status
  → Permission ... denied by the Claude Code auto mode classifier
  ```
  `git status`는 읽기 전용이라 그전까진 늘 자동 승인되던 명령이다. 게다가 `rtk gain`은 `No tracking data yet` — **재작성된 명령이 실행조차 안 됐다**는 뜻이라, "훅이 안 붙었나?"로 오해하기 쉬운 자리였다. 반면 PowerShell 도구로 돌린 같은 `git status`는 정상(exit 0)이었다.
- **원인**: **권한 심사는 내가 쓴 명령이 아니라 훅이 재작성한 최종 명령을 본다.** 순서가 `PreToolUse 훅 → 재작성 → 권한 판정 → 실행`이라, rtk가 `git status`를 `rtk git status`로 바꾼 뒤 분류기 앞에 선다. 분류기 입장에서 `rtk`는 **방금 인터넷에서 받은, 규칙에 없는 미지 바이너리**이므로 auto 모드에서 하드 거부한다(프롬프트조차 안 뜬다). 즉 훅 자체는 정상 동작했고 **정상 동작했기 때문에** 막힌 것이다. PowerShell 도구가 멀쩡했던 건 훅 matcher가 `Bash`라 그 경로만 재작성되기 때문 — 이 비대칭이 "훅 문제"라는 축을 오히려 가렸다.
- **해결**: 훅·허용 규칙·rtk 설정 **세 개를 한 세트로** 맞춘다.
  1. `~/AppData/Roaming/rtk/config.toml`의 `[hooks] exclude_commands`로 **쓰기 명령을 재작성에서 뺀다**(`git add`/`commit`/`push`/`worktree`, `gh pr create`/`merge`). 변경 작업에 서드파티 프록시를 끼우지 않고, 그 명령들은 종전대로 개별 심사된다.
  2. `.claude/settings.local.json`의 `allow`에 `Bash(rtk:*)`를 넣어 재작성된 읽기 명령을 통과시킨다.
  3. 같은 파일 `deny`에 rtk의 **임의 실행 서브커맨드**를 막는다 — `rtk run`(`sh -c` 그대로)·`proxy`·`err`·`summary`·`test`. 이게 없으면 `Bash(rtk:*)` 한 줄이 "아무 명령이나 자동 승인"이 된다. 훅은 이 다섯으로는 **재작성하지 않으므로**(`npm run test`→`rtk npm run test`, `cargo test`→`rtk cargo test`로 전용 필터에 붙는다) 막아도 기능 손실이 0이다.
- **재발방지**: **명령을 재작성하는 훅을 넣을 땐 "재작성 결과가 권한 규칙에 존재하는가"를 같이 확인한다** — 훅 설치와 허용 규칙은 별개 작업이 아니라 한 세트다. 넓게는 *중간 계층이 값을 바꾸면 그 값을 검사하는 하류 계층의 전제도 같이 깨진다*는 얘기고, 검증은 재작성 **전** 명령이 아니라 **후** 명령으로 해야 한다(`rtk hook check <명령>`이 그 확인 수단). 그리고 **한 도구를 통과시키려 넓은 allow를 여는 순간 그 도구의 임의 실행 경로가 곧 우회로**가 되므로, `Bash(<도구>:*)` 형태를 열 때는 그 도구의 서브커맨드 목록에서 "임의 명령을 실행해 주는 것"을 먼저 찾아 deny로 닫는다. 계열: [T-055](T-055.md) — 게이트와 실행이 서로 다른 것을 보고 있어 생긴 비대칭.
