# T-044 · vitest 설정이 없어 `.claude/worktrees/`의 버려진 워크트리 테스트까지 돌아, 세션 내내 green 숫자가 2.3배 부풀려짐

- **증상**: 커밋 메시지·PR 본문에 "vitest **412 / 427 / 444 / 466 green**"을 계속 적었는데, 실제 이 레포의 테스트는 **12파일 220건**이다. vitest는 **33파일 510건**을 돌리고 있었다. `npx vitest run src/`로 필터해도 **똑같이 510**이 나온다(워크트리 경로에도 `src/`가 들어 있어 필터가 안 먹는다). 전부 green이라 **아무 신호도 없었다** — 무성 실패.
- **원인**: `vitest.config.ts`가 **아예 없었다.** 설정이 없으면 vitest 기본 include(`**/*.test.ts`)가 레포 전체를 훑고, 기본 exclude는 `node_modules`·`dist`만 막는다 — **`.claude/`는 아무도 안 막아준다.** 그 안에 이전 세션들이 남긴 워크트리 2개(`ai-usage-eval-criteria-16c64d` detached / `nostalgic-booth-fd0849` on `docs/research-p7-lint`)가 살아 있었고, 각각 자기 트리의 **옛 `src/`를 import해 옛 테스트**를 돌렸다. 그 green은 **main에 대해 아무 말도 하지 않는다.**
- **해결**: `vitest.config.ts` 신설 — `include: ['src/**/*.{test,spec}.{ts,tsx}']` + `exclude: ['**/node_modules/**', '**/.claude/**', '**/.next/**']`. 220건으로 정상화. ⚠️ **워크트리 자체는 안 지웠다** — 내가 만든 게 아니고 이전 세션의 작업이 들어 있을 수 있다.
- **재발방지**: **테스트 러너의 "N green"을 인용하기 전에 N이 이 레포의 테스트 수인지 한 번은 확인한다** — `it(` 선언 수를 세어 대조하면 즉시 드러난다(`grep -rc "it(" src/**/*.test.ts` = 220 vs 리포터 510). 위험한 건 숫자가 틀린 것 자체가 아니라 **신뢰의 희석**이다: (1) 커밋 메시지에 박힌 "510 green"은 나중에 **아무도 검증 못 한다**(그때 워크트리가 몇 개였는지 모르니 재현 불가), (2) 워크트리의 낡은 테스트가 실패하면 **멀쩡한 작업이 막히고** 원인 추적에 시간이 든다. ⚠️ **T-030과 같은 계열** — 거기선 워크트리에 `node_modules`가 없어 링크가 깨진 것처럼 오진됐고, 여기선 워크트리가 **있어서** 테스트가 부풀려졌다. **워크트리 잔재는 도구가 레포 전체를 훑을 때마다 오염원이 된다** → 새 도구(테스트 러너·린터·링크 체커)를 붙일 땐 `.claude/`를 먼저 제외한다. 관련: 메모 [[vitest-no-typecheck-tsc-gate]] — 그건 vitest가 **뭘 안 보는가**였고, 이건 vitest가 **뭘 더 보는가**다. 둘 다 "green"의 의미를 흔든다.
