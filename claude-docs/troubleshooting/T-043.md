# T-043 · 콜백에 인자를 추가해도 **인자 적은 기존 호출부는 tsc가 통과시킨다** — 새 인자가 조용히 버려짐

- **증상**: `decide(state, accept, withWorkup)`에 3번째 인자를 추가하고 `ReceivingPhase`의 prop 타입도 `onDecide: (accept: boolean, withWorkup?: boolean) => void`로 넓혔는데, **「검사를 추가한다」 버튼을 눌러도 아무 일이 안 일어났다.** `tsc --noEmit` **0건**, vitest **444 green**. 순수 로직 테스트는 `decide()`를 직접 호출해서 전부 통과했다 — 끊긴 건 **UI 배선 한 줄**이다.
- **원인**: `SessionClient.tsx`가 `onDecide={(accept) => decide(s.receiving!, accept)}`였다. TypeScript는 **인자를 덜 받는 함수를 더 받는 함수 타입에 할당하는 걸 정상으로 본다**(파라미터 개수 이변성 — `(a) => void`는 `(a, b) => void`에 할당 가능하고, 이건 버그가 아니라 `arr.map(x => x)` 같은 관용구를 살리려는 의도된 설계다). 그래서 **넓힌 시그니처와 안 넓힌 호출부가 타입상 완벽히 정합**이고, 새 인자는 `undefined`로 흘러 기본값 `false`가 된다 — 에러도 경고도 없는 **무성 실패**.
- **해결**: 호출부를 `onDecide={(accept, withWorkup) => ... decide(s.receiving!, accept, withWorkup)}`로 고쳤다. 브라우저에서 버튼을 실제로 눌러보고 잡았다.
- **재발방지**: **콜백/핸들러 시그니처에 인자를 추가하면, 그 prop을 넘기는 호출부를 `grep`으로 전수 확인한다** — 컴파일러는 안 알려준다. 특히 **"순수 함수는 TDD로 잠그고 UI는 브라우저로 검증한다"는 이 프로젝트 구조에서 배선 한 줄이 정확히 두 그물 사이로 빠진다**(vitest는 `decide()`를 직접 부르고, tsc는 이변성으로 통과시킨다). ⚠️ 메모 [[vitest-no-typecheck-tsc-gate]]의 **역방향 사례**다 — 거기선 vitest가 놓친 걸 tsc가 잡았고, 여기선 **tsc·vitest 둘 다 놓친 걸 브라우저가 잡았다**. 새 선택지·새 인자를 UI에 붙였으면 **반드시 그 버튼을 눌러본다**. "테스트 green + tsc 0"은 배선이 됐다는 증거가 전혀 아니다.
