# T-050 · 훅이 막는 건 stale 기반 "쓰기"지 stale 기반 "사고"가 아니다

- **증상**: 워크트리에서 세션을 재개해 `plan.md`를 읽고 "완료 / 앞으로 할 일 / 살아있는 제약" 3층으로 분류한 뒤, **F4(가산금 배분)를 "앞으로 할 일 6줄"로 분류해 사용자에게 제시**했다. 그런데 `origin/main`에는 **F4가 이미 절단(#44)**돼 있었다 — 워크트리가 1커밋 behind였다. 낡은 `plan.md`를 최신으로 착각하고 분석·제시까지 끝낸 뒤에야 `git fetch`로 알았다. **`block-stale-base` 훅(글로벌 PreToolUse)은 한 번도 안 걸렸다.**
- **원인**: `block-stale-base`는 **첫 Write/Edit 시점**에 stale이면 deny하는 **쓰기 가드**다([T-038](T-038.md)에서 승격). 그런데 stale base의 비용은 "낡은 코드 위 재작업"만이 아니다 — **읽기·분류·계획·사용자 대면 제시**도 stale 입력이면 오염되는데, 이건 전부 쓰기가 아니라 **훅의 사각지대**다. T-038이 "낡은 코드 위에 다시 짠다"를 봤다면, 이건 **"낡은 상태 위에서 사고한다"** — 한 겹 위다. 특히 **컴팩션·세션 재개 직후**는 첫 행동이 대개 "읽고 분류"라 이 사각에 정확히 빠진다.
- **해결**: `git checkout -b <새브랜치> origin/main`으로 base를 최신화하고 **분류를 처음부터 다시** 했다 — F4는 "절단됨"으로 재분류. 실제 편집·커밋은 최신 base에서만. 오분류로 사용자에게 이미 제시한 건 **정정 고지**했다(잘못된 판단을 조용히 덮지 않는다).
- **재발방지**: **워크트리에서 세션을 재개하면, 읽기·분석보다 먼저** `git fetch && git rev-list --left-right --count origin/main...HEAD`로 base 신선도를 확인한다 — 훅에 의존하지 않는다(훅은 쓰기 가드지 읽기 가드가 아니다). ⚠️ **"훅이 안 걸렸다 = 안전하다"가 아니다** — 훅의 트리거(편집)를 안 지나가는 작업(분류·계획·리뷰)은 훅이 보증하지 않는다. **계열**: [T-038](T-038.md)(stale 기반 *쓰기* · 훅으로 승격) → **T-050**(stale 기반 *사고* · 훅 사각). 입력이 틀리면 분석을 정교화해도 **정교하게 틀릴** 뿐이라는 점에서 [[wrong-question-scales-with-agents]]와 같은 뿌리다.
