# T-051 · 워크트리 preview는 메인 레포에서 돈다 — 시각검증은 워크트리 직접 dev

- **증상**: 워크트리(`.claude/worktrees/*`)에서 layout.tsx·globals.css를 고쳐 커밋했는데, `preview_start {name:"dev"}`(포트 3000)로 열어도 화면에 반영이 안 됨. `document.documentElement.className`이 여전히 `geist_…variable`, `getComputedStyle(document.body).fontFamily`가 `Arial`, 새 CSS 변수(`--font-pretendard` 등)가 빈 값. **리로드해도 동일** — 즉 stale 브라우저가 아니라 서버가 다른 코드를 컴파일 중.
- **원인**: 메인 레포와 워크트리 양쪽에 `package-lock.json`이 있어 Next/Turbopack의 **워크스페이스 루트 추론이 메인 레포의 락파일 디렉터리를 루트로 선택**한다(dev 로그의 `⚠ Detected additional lockfiles`로 확인 가능). `preview_start`가 그 추론된 루트(=메인 레포)에서 `next dev`를 띄우므로, **워크트리엔 `.next`가 안 생기고 메인 레포의 구버전**(아직 Geist인 layout 등)을 서빙한다. 워크트리의 커밋·게이트(tsc/vitest)는 정상인데 **화면만 다른 코드를 본다.**
- **해결**: 워크트리 변경을 시각검증하려면 **워크트리에서 직접** dev 서버를 띄운다 — `cd <worktree> && npx next dev -p 3001`(다른 포트)을 백그라운드로, 브라우저를 그 포트로 navigate. 폰트·CSS 실측은 `document.fonts`(loaded 여부) + 렌더 폭 비교 + `document.fonts.check('100px "<family>"', '환자')`(한글 코드포인트 커버)로. (실측 2026-07-18: 이렇게 하니 Pretendard·D2Coding·Noto Serif KR 3계층 모두 한글 커버 확인.)
- **재발방지**: 워크트리 시각검증은 `preview_start`를 믿지 말고 **워크트리에서 직접 dev 서버**를 원칙으로. "dev 서버는 Bash 말고 preview_start" 기본 규칙의 **정당한 예외**(preview_start가 워크트리를 못 잡는 케이스). 첫 화면이 예상과 다르면 브라우저 캐시를 의심하기 전에 **어느 디렉터리에서 서버가 도는지**(`.next` 위치·dev 로그의 lockfile 경고)를 먼저 본다. 관련 메모리: [[worktree-preview-runs-in-main-repo]].
