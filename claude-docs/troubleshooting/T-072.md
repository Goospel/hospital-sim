# T-072 · CSS 청크 이름이 안 바뀌어 dev 서버가 옛 스타일을 계속 내줬다

- **증상**: `globals.css`에 `.fade-in`(keyframes + 클래스 + reduced-motion 오버라이드)을 추가하고 `Landing.tsx`가 그 클래스를 달았는데, 브라우저에서 `getComputedStyle`이 `animation-name: none`을 냈다. 클래스는 DOM에 정확히 붙어 있었고(`fade-in flex flex-col items-center`), 파일에도 규칙이 그대로 있었다. `document.styleSheets`를 전수 훑으니 같은 파일의 **기존** 규칙(`.hm-wander`)은 나오는데 **새 규칙만** 없었다. dev 서버를 껐다 켜도 동일했다.

- **원인**: Turbopack의 영속 캐시(`.next/`)가 CSS 청크를 재생성하지 않았다. 결정적 단서는 **청크 URL이 내용 변경 전후로 완전히 동일**했다는 것이다 — `/_next/static/chunks/[root-of-the-server]__0rp9myk._.css`. 파일명에 내용 해시가 안 붙는 개발 모드 청크라, 캐시가 "이 청크는 이미 만들어져 있다"고 판단하면 **내용이 달라도 이름이 같아 아무도 stale임을 눈치채지 못한다.** 브라우저 캐시가 아니라(‌`cache: no-store`로 받아도 옛 내용) 서버가 옛 산출물을 그대로 내주고 있었고, `npm run dev` 재시작은 `.next`를 안 지우므로 증상이 그대로 살아남았다.

  진단이 늦어진 이유: 증상이 **CSS 문법 오류나 Tailwind 레이어 우선순위처럼 보였다.** 같은 파일의 다른 규칙은 멀쩡히 적용되고 있어서 "파일은 읽히는데 내 규칙만 무시된다" → 내 CSS가 틀렸다는 가설로 먼저 갔다. 실제로는 파일 자체가 안 읽히고 있었다.

- **해결**: dev 서버를 끄고 `rm -rf .next` 후 재시작. 곧바로 서빙되는 CSS에 `fade-in`이 4회 등장했고 애니메이션이 정상 재생됐다(`playState: finished`, `currentTime: 700`).

- **재발방지**: **"내 CSS 규칙이 안 먹는다"의 첫 확인은 브라우저가 아니라 서빙되는 파일이다.** DOM·computed style·styleSheets를 아무리 파도 "누가 이 규칙을 지웠나"만 나오지 "이 규칙이 애초에 전송됐나"는 안 나온다. 한 줄로 끝난다:

  ```bash
  curl -s "http://localhost:3000$(curl -s http://localhost:3000 | grep -o '/_next/static/[^"]*\.css' | head -1)" | grep -c my-new-class
  ```

  0이면 코드 문제가 아니라 빌드 문제다 — 그때는 `rm -rf .next`가 먼저다. **파일명이 같다는 사실이 곧 stale을 숨기는 메커니즘**이므로, URL 해시가 안 바뀐 걸 "변경 없음"의 증거로 읽지 않는다(오히려 그게 의심 신호다).

  계열: [T-065](T-065.md)(도구가 얼어붙은 채 정직하게 오답을 냈다)와 같은 모양이다 — 거기선 스크린샷 도구가, 여기선 dev 서버가 **에러 없이 옛 값을 성실히** 내줬다. 둘 다 "출력이 있다 = 최신이다"를 무의식적으로 가정한 데서 왔다.
