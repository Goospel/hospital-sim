# T-048 · git 훅은 스크립트를 직접 실행해서 검증되지 않는다 — 실제 `git commit`으로만 드러나는 게 셋 있다

- **증상**: `pre-commit`(sh)이 PowerShell 검사기를 부르게 만들고, 검사기 자체는 TDD로 **22/22 green**이었다. 그런데 실제 `git commit`을 하니 **세 번 연속 다르게 실패**했다. ① 1차: 커밋이 **그냥 통과**했다(훅이 아예 안 돎 — 가장 나쁜 실패). ② 2차: 거부는 됐는데 사유가 *"목차가 stale하다"*였다. 실제 원인은 stale이 아니라 **스크립트를 로드조차 못 한 것**이었다 — 인덱스가 최신이어도 똑같이 거부했을 것이다. ③ 3차: 이번엔 스크립트가 멀쩡히 돌았는데 훅이 *"검사가 실행되지 않았습니다"*라고 오판했다.
- **원인**: 셋 다 **PowerShell 도구/콘솔에서 스크립트를 직접 실행하면 재현되지 않는다.** ① `core.hooksPath`가 `.git/worktrees/<name>/config.worktree`에 **절대경로**(메인 워크트리의 `.githooks`)로 박혀 있었다. 워크트리 스코프가 local 스코프를 이기므로, **워크트리 안에서 새로 만든 훅은 git 눈에 존재하지 않는다.** 기존 `commit-msg`는 메인 레포에 있어 정상 작동했고, 그래서 "훅은 잘 돌고 있다"는 착시가 있었다 — 트레일러 검사는 통과했으니까. ② Git Bash에서 `powershell.exe -File`을 부르면 기본 ExecutionPolicy(Restricted)가 걸려 **로드 자체가 안 된다**. ③ PowerShell 5.1이 콘솔로 내보내는 한글은 CP949라 bash에서 깨진다(`목차` → `������`). 훅이 한글로 상태를 판정하고 있어서 매칭이 실패했다 — **글로벌 규칙 2(인라인 인자의 한글 CP949)의 출력 방향 판**이다.
- **해결**: ① `git config --worktree core.hooksPath .githooks` — **상대경로**로 바꾸면 각 워크트리가 자기 `.githooks`를 본다(`commit-msg` 훅 주석이 원래 안내하던 형태다). ② 훅에서 `powershell.exe -NoProfile -NonInteractive -ExecutionPolicy Bypass -File …`. ③ 스크립트의 **모든 종료 경로**가 `INDEX-CHECK: OK|STALE|INVALID|REBUILT` 같은 **ASCII 마커**를 출력하게 하고, 훅은 그 마커로만 판정한다. 사람이 읽는 문장은 한글로 두되 기계 판정은 ASCII로 분리한다.
- **재발방지**: **git 훅은 반드시 실제 `git commit`으로 검증한다.** 스크립트를 직접 실행하는 테스트는 훅의 세 층(git이 훅을 찾는가 / 셸이 인터프리터를 띄우는가 / 출력이 셸까지 온전히 오는가)을 **하나도 건너지 않는다** — 검사기 22/22 green이 위 셋 중 아무것도 못 막았다. 검증 순서: (1) **거부돼야 하는 상태로 실제 커밋** → 거부되는가, (2) **거부 사유가 진짜 그 사유인가**(이유가 틀리면 다음에 원인을 엉뚱한 데서 찾는다), (3) 그다음 통과 케이스. ⚠️ **T-045와 같은 계열이되 한 층 위다** — 거기선 테스트가 fail-open을 못 봤고, 여기선 **테스트가 훅 경로 자체를 안 지나갔다.** 그리고 ①은 이 레포의 워크트리 함정 계열([T-028](T-028.md)·[T-030](T-030.md)·[T-038](T-038.md)·[T-044](T-044.md))에 하나 더 얹는다: **워크트리는 파일뿐 아니라 `config.worktree`로 설정까지 갈라놓는다.** 새 훅을 추가할 땐 `git config --show-origin --get-all core.hooksPath`로 **어느 파일의 어느 값이 이기는지** 먼저 본다.
