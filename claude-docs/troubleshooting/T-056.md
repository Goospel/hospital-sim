# T-056 · exhaustive switch union 확장과 case 배선을 다른 태스크로 나누면 태스크당 tsc 게이트가 깨진다

- **증상**: 성장 시스템 SDD 중 Task 5(`SessionPhase`에 `'GROWTH'` 추가) 커밋에서 **프로젝트 전체 `npx tsc --noEmit`가 red** — `SessionClient.tsx`의 `switch (session.phase)` 끝 `default: { const _exhaustive: never = session.phase; return _exhaustive; }`에서 "`'GROWTH'`는 `never`에 대입 불가". 정작 Task 5가 만진 `session.ts`/`session.test.ts` **자체는 tsc clean**인데 프로젝트 tsc는 깨진다. 계획은 SessionClient 배선을 **Task 9**로 미뤄, 방치하면 Task 5~9 **4개 태스크 동안** 게이트가 red인 채로 진행된다.
- **원인**: discriminated union(`SessionPhase`)에 값을 더하면, 그 union을 `default`에서 `const _x: never = value`로 **소진 검사(exhaustive never)**하는 switch가 **즉시** 컴파일 에러가 된다(새 값이 아직 case로 안 닫혀 `never`에 안 들어감). 그 switch의 새 case를 **나중 태스크**에서 배선하도록 계획이 순서를 잡으면, union-확장 태스크와 case-배선 태스크 **사이의 모든 커밋**에서 프로젝트 tsc가 깨진다. 태스크당 `tsc --noEmit` 게이트는 vitest(esbuild)가 못 잡는 타입 회귀를 잡는 **유일 게이트**([vitest-no-typecheck 계열])인데, 그 구간에서 무력화된다. 서브에이전트는 지시(“SessionClient 건드리지 마라, Task 9 몫”)를 지켜서 오히려 게이트가 깨진 채로 DONE을 반환했다 — **스코프 규율과 게이트 규율이 충돌**한 것.
- **해결**: union-확장 태스크(Task 5)에서 그 switch에 **최소 스텁 case**를 같이 넣어 exhaustive 검사를 즉시 만족시킨다 — `case "GROWTH": return null;`(아직 런타임 미도달이라 무해). 실제 렌더 배선은 나중 태스크가 스텁을 교체한다([src/components/SessionClient.tsx](../../src/components/SessionClient.tsx), ae5859b 스텁 → 5150441 배선). 컨트롤러가 리뷰 전에 tsc 결과에서 감지해 스텁 fix를 즉시 지시했다(구현자 보고가 “tsc 1 error, 스코프 밖이라 안 고침”으로 정직하게 올라온 게 감지 지점).
- **재발방지**: 계획을 쪼갤 때, **discriminated union에 값을 추가하는 태스크는 그 union을 `never`로 소진 검사하는 모든 switch의 case도 같은 태스크에서(최소 스텁으로라도) 닫아라.** case 배선(렌더 로직)은 나중 태스크로 미뤄도 되지만, **case의 존재 자체는 union 확장과 같은 커밋**이어야 태스크당 tsc 게이트가 중간 구간 내내 green이다. 계획 pre-flight 체크 한 줄: *"union을 확장하는 태스크가 그 union의 exhaustive switch도 같은 태스크에서 닫나?"* 넓게는 — 타입 레벨 전수 검사(exhaustive never)는 **union 확장과 그 소비를 같은 커밋에 묶어야** 게이트가 안 깨진다(스코프 최소화보다 게이트 무결성이 우선하는 드문 경우).
