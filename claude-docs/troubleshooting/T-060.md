# T-060 · Pages 배포 워크플로가 자기 자신의 전제를 만들 수 없다 — 첫 배포 2연속 실패

- **증상**: `.github/workflows/deploy-pages.yml`을 main에 머지하자 첫 실행이 죽었다.
  ```
  ##[error]Get Pages site failed. Please verify that the repository has Pages
  enabled and configured to build using GitHub Actions, or consider exploring
  the `enablement` parameter for this action.
  HttpError: Not Found
  ```
  에러가 친절하게 `enablement` 파라미터를 가리키길래 `enablement: true`를 넣고 다시 돌렸다. **두 번째도 죽었다** — 이번엔 다른 이유로.
  ```
  ##[error]Create Pages site failed. Error: Resource not accessible by integration
  ```
  빌드·테스트·export는 두 번 다 통과했다. 죽은 건 언제나 `configure-pages` 스텝 하나였다.
- **원인**: **Pages 사이트 생성은 `GITHUB_TOKEN` 권한 밖이다.** 워크플로에 `permissions: pages: write`를 줘도 그건 *이미 있는 사이트에 배포할* 권한이지 *사이트를 만들* 권한이 아니다(후자는 `administration` 계열이라 자동 토큰에 안 실린다). 즉 **워크플로가 자기 자신이 돌기 위한 전제를 만들 수 없는 부트스트랩 구조**다. 액션의 에러 메시지가 `enablement` 파라미터를 권하는 게 함정을 키운다 — 그 옵션은 조직 정책·PAT 환경에서만 통하는데 메시지는 그 조건을 말하지 않아, **문서가 가리키는 해결책을 그대로 따랐는데 다른 에러로 죽는** 경로가 만들어진다.
- **해결**: 소유자 권한으로 **1회** 켠다. 그러면 그 뒤 모든 실행은 그냥 통과한다.
  ```bash
  gh api -X POST repos/<owner>/<repo>/pages -f build_type=workflow
  ```
  응답의 `html_url`이 배포될 주소다(`build_type: workflow` 확인). 웹 UI로는 Settings → Pages → Source를 **GitHub Actions**로. 켠 뒤 실패한 런을 `gh run rerun <id>`로 다시 돌리면 된다(재커밋 불필요). `enablement: true`는 아무것도 해결하지 못하므로 **워크플로에서 제거하고**, 대신 이 절차를 주석으로 남겼다 — 옵션을 남겨두면 "이게 켜주는구나"라는 오해가 남는다.
- **재발방지**: **CI가 외부 리소스를 "설정"하는 스텝은 그 리소스가 이미 존재한다고 가정한다** — 새 레포/새 fork에서 배포 워크플로를 처음 돌릴 땐 *워크플로 밖에서 1회 해야 하는 수동 단계*가 있는지 먼저 본다(Pages 활성화, 환경 시크릿, 배포 환경 승인 규칙 등). 그리고 **에러 메시지가 제안하는 파라미터를 권한 조건 확인 없이 따르지 않는다**: 여기서는 제안대로 고친 뒤 *같은 스텝이 다른 에러로* 죽어, 두 번째 실패를 볼 때까지 "권한 문제"라는 축이 안 보였다. 넓게는 [T-045](T-045.md)·[T-046](T-046.md) 계열 — **검사·설정이 "돌지 않았을 뿐인데 통과처럼 보이는" 자리**를 조심한다(여기선 반대로 시끄럽게 죽어서 다행이었다).
