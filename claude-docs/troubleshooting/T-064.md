# T-064 · 좌표가 공식과 맞는 걸 확인했지, 대상과 맞는 걸 확인하지 않았다

- **증상**: 병원 맵 슬라이스(PR #98)의 브라우저 실측이 **9개 항목 전부 통과**로 보고됐다. 스크린샷 compositing이 이 환경에서 막혀(`the Browser pane is not displayed, so the page is not compositing frames`) `javascript_tool`로 DOM의 `className`·`left`·`top`을 직접 질의하는 방식으로 대체했고, 수치는 전부 정확했다 — 환자 아바타 `top:84%`, 병상 `left:16.6667%`, 의사 `top:32.24%`가 코드의 공식과 소수점까지 맞았다. **그런데 환자가 침대 위에 없었다.** 데스크톱 폭에서 침대 아래로 65px 떨어진 자리에 회색 인물들이 한 줄로 떠 있었다. 「수용한 환자가 **침대 위에** 있게 한다」가 이 슬라이스 설계의 명시 목표(spec §1)인데 그게 안 된 채로 실측이 초록이었다. 최종 전 브랜치 리뷰가 CSS에서 손계산으로 잡았다.
- **원인**: 검증이 **공식 내부의 일관성**만 확인하고 **공식과 대상의 관계**를 확인하지 않았다. 환자 `top`은 `ROOMS_H + CORRIDOR_H + 16` = 84%였고 실측값도 84%였다 — 일치. 하지만 침대 중심은 퍼센트가 아니라 **px 스택**으로 정해진다: 병동 밴드 상단 `0.68H` + `pt-1.5`(6px) + 침대 `sm:h-9`의 절반(18px) = `0.68H + 24px`. 환자는 `0.84H`. 차이 `0.16H − 24px`는 `H = 150px`(맵 폭 ~267px)에서만 0이고, 실제 데스크톱(`max-w-5xl` → 맵 992px → H=558)에선 65px이다. 즉 **`16`은 어느 화면에서도 맞은 적이 없는 값**이었다 — 계획 단계에서 손으로 찍었고, 태스크 3 리뷰는 그걸 "손튜닝인데 주석이 없다"는 Minor로만 처리해 **설명을 붙이고 값은 재지 않았다.**

  구조적으로는 **태스크 경계를 가로지르는 결함**이었다: 태스크 2가 스프라이트 박스(`h-7 sm:h-9`)를 정하고 태스크 3이 오프셋을 정했는데, 둘을 대조한 태스크가 없다. 태스크별 리뷰는 각자의 diff만 보므로 구조상 볼 수 없었다.
- **해결**: `top`을 맵 높이의 퍼센트에서 **스프라이트 박스를 따라가는 `calc()`**로 바꿨다 — `top: calc(${ROOMS_H + CORRIDOR_H}% + 1.5rem)`. 1.5rem(24px) = `pt-1.5`(6px) + `sm:h-9` 절반(18px). 퍼센트만으로는 px로 정의된 스택을 표현할 수 없어 **어떤 상수를 넣어도 특정 폭에서만 맞는다** — `calc()`가 필수였다. 주석에 침대 스프라이트 크기와의 의존을 명시해, 스프라이트를 바꾸면 여기도 바꿔야 한다는 게 보이게 했다.

  검증은 **두 요소의 `getBoundingClientRect()` 세로 겹침**으로 다시 했다(데스크톱 1280×800·모바일 375×812 양쪽). 모바일에선 환자 bottom과 침대 bottom이 254.83으로 정확히 일치.
- **재발방지**: **UI 실측 항목은 "이 값이 공식대로인가"가 아니라 "이 요소가 저 요소와 어떤 관계인가"로 쓴다.** 전자는 코드가 코드대로 도는 것만 확인하고, 그건 언제나 참이다.

  구체적으로 — ⓐ **스크린샷이 막히면 rect 대조로 대체한다.** `getBoundingClientRect()` 둘을 비교하는 건 스크린샷보다 오히려 **강한** 증거다(수치가 남는다). 단 비교 대상을 **설계 문장에서** 뽑아야 한다: "환자가 침대 위에" → `patient.rect`와 `bed.rect`가 겹치는가이지, `top`이 84%인가가 아니다. ⓑ **손튜닝 상수를 만나면 설명이 아니라 측정을 요구한다** — 태스크 3 리뷰는 `16`에 주석이 없다고 지적했고 주석이 붙었지만, *설명된 틀린 값은 여전히 틀린 값이다.* ⓒ **퍼센트와 px가 섞이는 자리를 의심한다**: 컨테이너 비율(`aspect-*`)로 높이가 정해지는 레이아웃에서 자식이 px 크기를 가지면 퍼센트 오프셋은 **한 폭에서만** 맞는다 → `calc()`를 기본값으로. ⓓ **태스크 경계를 가로지르는 결함은 태스크별 리뷰가 구조상 못 본다** — 두 태스크가 각자 정한 값이 서로 맞아야 하는 관계(스프라이트 크기 ↔ 배치 오프셋)를 잡는 건 **최종 전 브랜치 리뷰의 고유 역할**이고, 그래서 그걸 생략하면 이 계열은 전부 샌다.

  계열: [T-063](T-063.md)이 **같은 모양**이다 — 거기선 검사기가 "파일이 디스크에 있나"를 봤지만 진짜 불변식은 "옵시디언 볼트에서 해석되나"였다. 검사가 실제로 돌았고 결과도 정직했으며, 다만 **틀린 질문에 정직하게 답했다.** 이번엔 실측이 "좌표가 공식대로인가"에 정직하게 답했고 진짜 질문은 "환자가 침대 위인가"였다. 대리 지표(proxy)를 검증하면 **대리 지표가 맞는 동안에는 초록이고, 어긋나는 자리에서만 조용히 틀린다.**
